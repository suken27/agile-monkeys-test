Good catch — passing the full list is the wrong call, and it actually contradicts the codebase's own stated design.

Look at PromotionRepositoryPort.java:12-16: the port's class javadoc says "Filtering for the invariants... is the implementing adapter's responsibility, not the caller's — the port exposes exactly the queries those invariants need, already scoped." But findByApplicationAndTargetEnvironment at line 25 is documented as "Every known promotion for this pair" — not scoped at all. Then Promotion.request at Promotion.java:79 does the !p.status.isTerminal() filtering itself. So the actual behavior contradicts what the port claims to guarantee.

There's a real design tension underneath:

Precedent for adapter-side filtering already exists: findLastCompletedEnvironment (port line 28) already returns an aggregate/filtered result, not raw rows. Terminal-ness is domain vocabulary (PromotionStatus.isTerminal()), so an adapter query like WHERE status NOT IN (COMPLETED, ROLLED_BACK, CANCELLED) isn't reinventing business logic — it's just consuming the existing enum. That would make findByApplicationAndTargetEnvironment (or a renamed findActivePromotionsForTarget) actually match its class-level contract, and it's more efficient — no need to drag in every historical completed/cancelled promotion just to check for one active conflict.
Counter-argument for keeping the filter in request(): Promotion's own class javadoc (lines 15-16) says it "guards every business rule... no repository, handler, or API layer can bypass an invariant enforced here" — i.e., don't trust the adapter to have filtered correctly. If the adapter's query is ever wrong, a defense-in-depth check in the domain catches it.
My recommendation: pre-filter in the adapter for efficiency (rename the port method to reflect that it returns only non-terminal promotions, and fix its javadoc), but keep a cheap invariant check in the domain — just simplify it to !existingPromotionsForTarget.isEmpty() rather than dropping the check entirely. That gets you both the efficiency win and the "domain doesn't blindly trust the adapter" property, without fetching data you'll never use.
